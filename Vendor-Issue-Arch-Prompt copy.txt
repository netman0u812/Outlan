2) Broker as policy enforcement point / service plane

This is what you mean by local. And you’re right: in a flat network, you

What you can do in a flat enterprise network (three realistic paths)
Path A — Accept edge brokering, but create app enclaves

This is the structural fix:

carve the flat network into app islands (VRF/VLAN/security zones/cloud VPCs)

deploy the enforcement plane (proxy/inspection) inside each island

now the access broker becomes “local” again because the network has locality

This is the cleanest way to reconcile your principle with reality.



Path B — Keep the network flat, but introduce a service plane overlay

This is basically RC1 logic:

“tenancy” is not the network; it’s the proxy + DNS + PKI service plane

you deliberately force all third-party traffic through that plane

locality is “local to the service plane” rather than “local to the app”

This works even when the enterprise is flat, because you are creating an artificial boundary.



Path C — Hybrid: Use an access broker for “who”, but keep bespoke service termination local

This is often the most practical compromise:

Access broker handles identity and session authorization

Your proxy/gateway fabric (RC1-style) is what actually terminates and enforces per-tenant service profiles

The broker never tries to be the service plane

This matches your intuition best: broker is not the proxy factory.




The key takeaway

You’re highlighting a structural truth:

In a flat enterprise network, “local-to-workload brokering” is not a property you can buy.

It’s a property you have to create by introducing boundaries:

app enclaves, or

a service plane overlay, or

both.

If you don’t change structure, the broker will always feel “non-local” because there is nowhere for “local” to attach to.
